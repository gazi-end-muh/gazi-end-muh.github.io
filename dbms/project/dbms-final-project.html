Project Overview

The aim of the project is to evaluate your understanding and application of database management systems concepts. This is an open-ended final project. You will form groups and each group will choose a real-world problem domain and build a complete database solution for that domain.

You are expected to physically create the database, populate it with realistic records, and write a report on your progress and work. This is a group project, so you should divide the work among your team members and collaborate effectively to produce a comprehensive final submission.

Group Size: You may work in groups of up to 4 students.

Deadline: Friday, 22 May 2026, 23:59 (Türkiye Local Time).

Submission Type: Submission-only project (no in-class presentation/demo required).

Delivery: Send your project files and report by email.

Suggested Project Domains (Examples)

  • Library, bookstore, or second-hand publication management
  • Course attendance and student activity tracking
  • Restaurant order and service management
  • Hotel or reservation management
  • Personnel, payroll, or shift management
  • Shipment, stock, or inventory tracking

What You Are Expected to Build (Step by Step)

  1. Project Selection & Requirement Analysis: Select a project topic and define the business scope, key actors, and data requirements. Conduct a comprehensive requirement analysis phase as if you were working on a real-world industry project, including interviews with field professionals if necessary.
  2. Conceptual Model Design: Develop a conceptual model (ER diagram) that clearly identifies entities, attributes, and relationships. Build your design using a professional tool like Lucidchart or provide a high-quality scan of a hand-drawn diagram.
  3. Logical Relational Schema Mapping: Map your conceptual design to a logical relational schema, ensuring all primary keys, foreign keys, and integrity constraints are defined. Build this design digitally or provide a clear, scanned version.
  4. Normalization: Apply normalization techniques (e.g., 1NF, 2NF, 3NF) where applicable to eliminate data redundancy and ensure database integrity.
  5. Physical Database Creation: Physically implement the database, including all tables, keys, constraints, and relationships. Save the complete DDL creation script for the final report and project delivery.
  6. Data Population & Consistency: Insert realistic sample records while maintaining strict data consistency across all tables. Save your DML data insertion script for the final report and project delivery.
  7. Advanced SQL Operations & Analytics: Implement at least 10 distinct SQL queries that demonstrate a deep mastery of the course material. These queries should produce meaningful analytical outcomes or demonstrate advanced features like complex joins, subqueries, and aggregations.
  8. Final Report & Documentation: Prepare a comprehensive final report in English, including all required design sections, appendices, and functional SQL scripts. Ensure all scripts execute correctly; any queries that fail will result in a grade reduction.

Functional Requirements

Part 1 - System Definition and Conceptual Design

  • Clearly define the problem, users, and system boundaries, document your design decisions and requirement analysis.
  • Produce ER model with correct entity/relationship semantics and cardinalities.
  • Show assumptions and business rules that drive your design decisions.

Part 2 - Logical Schema and Data Integrity

  • Convert ER model to relational schema with primary keys and foreign keys.
  • Use integrity constraints (NOT NULL, UNIQUE, CHECK, reference integrity).
  • Document normalization decisions and resulting table structures.

Part 3 - Physical Database Build and Population

  • Create all tables and relations physically in a DBMS environment, document your steps and queries for future creation and reporting.
  • Populate data with meaningful records for testing and analysis.
  • Show that your schema and data support real project scenarios.

Part 4 - Progressive Implementation of Weekly Course Topics

As new topics are taught, update your project and include evidence in your report. Expected coverage includes:

  • SELECT and nested queries
  • INSERT, UPDATE, DELETE operations
  • Variables, arithmetic/logical operators, control statements, loops
  • Error handling and transactions
  • Concurrency-related considerations
  • Views, triggers, user-defined functions, stored procedures
  • Indexes (and synonyms if used)

Important: Your final project should reflect the course content covered up to the submission date, not only early-week topics.

Usage of AI

  • Allowed: concept clarification, SQL syntax reminders, debugging hints, and planning support.
  • Not allowed: direct generation of full project schema, full SQL package, or full report as final submission.
  • If AI tools are used, declare tool name, date, and how outputs were verified/modified by your team.

Each group is responsible for understanding and defending every design and implementation decision.

Project Report

Prepare one final group report in English, write your group members' names and student numbers in the cover page with project name.
